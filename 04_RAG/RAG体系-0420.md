# RAG 知识体系

## 一、什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）**

### 为什么需要 RAG？

LLM 有两个天生缺陷：
- **训练数据有截止日期**：不知道最新发生的事情
- **无法接触私有数据**：公司内部文档没法用来训练模型

RAG 的解法：把私有文档存进向量数据库，用户提问时先检索相关文本块，再把问题 + 文本块一起交给 LLM 生成答案。

---

## 二、RAG 全流程

整体分两个阶段：**建库** + **问答**

```
【建库阶段】 文档加载 → 文档切块 → 向量化 → 存入向量数据库

【问答阶段】 查询优化 → 混合检索 → Reranking重排 → 组装Prompt → LLM生成 → RAGAS评估
```

---

## Step 1：文档加载（Document Loading）

**做什么：** 把各种格式的文件，统一转换成纯文本。

支持格式：PDF、Word、Excel、CSV 等

**工具选择：**

| 方式 | 工具 |
|------|------|
| 框架组件 | LangChain / LlamaIndex 的 Document Loaders |
| 开源库 | PyMuPDF（PDF）、python-docx（Word）、pandas（CSV） |
| 高阶处理 | OCR + LLM + Prompt，处理扫描件、复杂表格 |

**代码示例：**

```python
from langchain_community.document_loaders import (
    PyMuPDFLoader,       # PDF
    Docx2txtLoader,      # Word
    CSVLoader,           # CSV
    DirectoryLoader,     # 批量加载整个目录
    UnstructuredMarkdownLoader,  # Markdown
)

# 加载单个 PDF
loader = PyMuPDFLoader("docs/company_report.pdf")
documents = loader.load()
print(f"加载了 {len(documents)} 页")

# 批量加载目录下所有 PDF
loader = DirectoryLoader("docs/", glob="**/*.pdf", loader_cls=PyMuPDFLoader)
documents = loader.load()

# 每个 Document 包含：page_content（文本内容）+ metadata（来源、页码等）
for doc in documents[:2]:
    print(doc.page_content[:100])
    print(doc.metadata)
```

## Step 2：文档切块（Chunking）

**做什么：** 把长文本切成小块，因为 LLM 上下文窗口有限，不能把整本书全塞进去。

> 🗣️ **大白话：** 块太小 → 没上下文，答不好；块太大 → 噪音多，检索不准还浪费 Token。

### 四种切块策略

| 策略 | 说明 |
|------|------|
| **固定长度切块** | 按固定 Token 数/字符数切分，简单高效，比如每块 500 字 |
| **语义切块** | 按段落、章节、标题等语义边界切分，保持信息完整 |
| **父子切块（Parent-Child Chunking）** | 小块（256字）用于检索，命中后返回对应大块（1024字）给 LLM |
| **滑动窗口切块** | 固定长度基础上加重叠部分（如 100 字），避免切断关键信息，提升召回 |

**代码示例：**

```python
from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    MarkdownHeaderTextSplitter,
)

# ① 固定长度 + 滑动窗口（最常用）
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # 每块最大字符数
    chunk_overlap=50,     # 相邻块之间重叠 50 字，避免切断关键句
    separators=["\n\n", "\n", "。", "！", "？", " ", ""],  # 按优先级尝试切分
)
chunks = text_splitter.split_documents(documents)
print(f"切块后共 {len(chunks)} 个块")

# ② 按 Markdown 标题语义切块
headers_to_split_on = [
    ("#", "Header1"),
    ("##", "Header2"),
    ("###", "Header3"),
]
md_splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
md_chunks = md_splitter.split_text(markdown_text)

# ③ 父子切块：小块检索，大块喂给 LLM
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore

parent_splitter = RecursiveCharacterTextSplitter(chunk_size=1000)
child_splitter = RecursiveCharacterTextSplitter(chunk_size=200)

store = InMemoryStore()
retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,      # 小块存向量数据库
    docstore=store,               # 大块文档存储
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)
retriever.add_documents(documents)
```

## Step 3：向量化（Embedding）

**做什么：** 把文本块转成向量（一串数字），让计算机能做语义检索。

> 🗣️ **大白话：** 把"苹果"和"水果"都变成空间中的点，语义越接近的词，点的距离越近。

### 常用 Embedding 模型

| 模型 | 部署方式 | 特点 |
|------|----------|------|
| **BGE**（BAAI General Embedding，北京智源通用向量模型） | 本地 | 中文效果最好，免费，首选 |
| **text2vec** | 本地 | 轻量，适合资源有限的环境 |
| **OpenAI text-embedding-3** | API | 效果好，收费 |
| **Cohere Embed** | API | 多语言强，收费 |

> BGE 由北京人工智能研究院（BAAI，Beijing Academy of Artificial Intelligence）开源。

**代码示例：**

```python
# 方式一：本地 BGE 模型（推荐，免费）
from langchain_huggingface import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-large-zh-v1.5",   # 中文首选
    model_kwargs={"device": "cpu"},          # 有 GPU 改成 "cuda"
    encode_kwargs={"normalize_embeddings": True},  # 余弦相似度需要归一化
)

# 方式二：OpenAI API
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",  # 便宜版，text-embedding-3-large 更准
)

# 测试效果：语义相近的词，向量距离应该很近
import numpy as np

vec1 = embeddings.embed_query("苹果")
vec2 = embeddings.embed_query("水果")
vec3 = embeddings.embed_query("汽车")

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

print(f"苹果 vs 水果：{cosine_similarity(vec1, vec2):.4f}")  # 应该较高，如 0.85
print(f"苹果 vs 汽车：{cosine_similarity(vec1, vec3):.4f}")  # 应该较低，如 0.45
```

## Step 4：向量数据库（Vector Database）

**做什么：** 存储所有向量，并根据相似度快速检索。

### 相似度计算方式

| 方式 | 原理 | 越相似则 |
|------|------|----------|
| **余弦相似度（Cosine Similarity）** | 衡量两向量夹角 | 值越接近 1 |
| **欧氏距离（Euclidean Distance）** | 衡量两向量之间的直线距离 | 值越小 |
| **内积（Dot Product）** | 两向量的点积 | 值越大 |

### 常用向量数据库

| 数据库 | 适用场景 |
|--------|----------|
| **ChromaDB** | 学习 / 原型开发，易上手 |
| **PGVector** | 已有 PostgreSQL，直接扩展，减少系统复杂度 |
| **Milvus** | 大规模数据，支持分布式，性能强但配置复杂 |
| **Qdrant** | 生产环境，高维向量，性能优越 |

**代码示例：**

```python
# 方式一：ChromaDB（本地，适合开发）
from langchain_chroma import Chroma

# 建库：把 chunks + embeddings 写入数据库（持久化到本地目录）
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",  # 持久化存储路径
    collection_name="my_docs",
)

# 加载已有库
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
    collection_name="my_docs",
)

# 方式二：PGVector（PostgreSQL 扩展，适合生产）
from langchain_postgres import PGVector

CONNECTION_STRING = "postgresql+psycopg://user:password@localhost:5432/rag_db"

vectorstore = PGVector.from_documents(
    documents=chunks,
    embedding=embeddings,
    connection=CONNECTION_STRING,
    collection_name="my_docs",
    use_jsonb=True,  # 元数据用 JSONB 存储，查询更快
)

# 相似度检索
query = "公司的退款政策是什么？"
results = vectorstore.similarity_search(query, k=5)   # 返回最相关的 5 个块
# 带分数的检索
results_with_score = vectorstore.similarity_search_with_score(query, k=5)
for doc, score in results_with_score:
    print(f"相似度: {score:.4f} | 内容: {doc.page_content[:80]}")
```

## Step 5：查询优化（Query Optimization）

**为什么要优化：** 用户提问往往口语化、模糊，直接检索效果差。

> 🗣️ **大白话：** 用户说"RAG效果不好咋整"，直接搜这句话检索不到什么，但改成"RAG检索准确率的影响因素及优化方法"就能搜到很多有用内容。

### 三种优化技巧

**① Query Rewriting（查询重写）**

用规则或模型将用户问题改写成更适合检索的表达。

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

rewrite_prompt = ChatPromptTemplate.from_template("""
你是一个搜索优化助手。请将用户的口语化问题，改写为更适合文档检索的专业表达。
只输出改写后的查询语句，不要解释。

原始问题：{question}
改写后的查询：
""")

llm = ChatOpenAI(model="gpt-4o-mini")
rewriter = rewrite_prompt | llm

result = rewriter.invoke({"question": "RAG效果不好咋整"})
print(result.content)  # → "RAG 检索准确率低的常见原因及优化方法"
```

**② HyDE（Hypothetical Document Embedding，假设文档向量化）**

让 LLM 先根据问题生成一段"假设的答案文档"，再对这段假设文档做 Embedding 检索。

> 🗣️ **大白话：** 先"脑补"一个答案，用这个答案去找资料，比直接用问题找更准。

```python
hyde_prompt = ChatPromptTemplate.from_template("""
请根据下面的问题，生成一段假设的参考文档内容（约100字），
即使你不确定答案，也请尽量生成相关内容。

问题：{question}
假设文档：
""")

# 用假设文档的向量去检索，而不是用问题本身
hyde_chain = hyde_prompt | llm
hypothetical_doc = hyde_chain.invoke({"question": "公司的加班政策是什么？"})

# 用假设文档的向量做检索
results = vectorstore.similarity_search(hypothetical_doc.content, k=5)
```

**③ 多查询并行（Multi-Query）**

针对同一问题从多个角度改写，并行检索，合并结果，提升召回率。

```python
from langchain.retrievers.multi_query import MultiQueryRetriever

retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

multi_query_retriever = MultiQueryRetriever.from_llm(
    retriever=retriever,
    llm=llm,
    # 默认会让 LLM 从 3 个不同角度改写问题，并行检索后合并去重
)

docs = multi_query_retriever.invoke("RAG 和微调怎么选？")
print(f"多查询检索到 {len(docs)} 个不重复文档块")
```

## Step 6：混合检索（Hybrid Retrieval）

**做什么：** 结合向量检索（语义）和关键词检索（精确），取两者之长。

| 检索方式 | 优点 | 缺点 |
|----------|------|------|
| **向量检索**（语义检索） | 理解语义，能处理模糊问题 | 精确词匹配弱 |
| **关键词检索**（如 BM25） | 精确匹配关键词 | 不理解语义 |
| **混合检索** | 结合两者，召回更全 | 需要合并排序 |

> 🗣️ **大白话：** 一个靠"意思"找，一个靠"关键词"找，两种一起用，才能又准又全。

**代码示例：**

```python
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

# 向量检索器
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# BM25 关键词检索器（需要所有文档的原始文本）
bm25_retriever = BM25Retriever.from_documents(chunks)
bm25_retriever.k = 10

# 混合：EnsembleRetriever 用 RRF（Reciprocal Rank Fusion，倒数排名融合）合并结果
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6],  # 关键词权重 0.4，向量权重 0.6，可按需调整
)

docs = ensemble_retriever.invoke("公司 2023 年营收增长率")
print(f"混合检索召回 {len(docs)} 个文档块")
```

## Step 7：重排序（Reranking）

**为什么要重排：** 混合检索召回了 Top 20，但排第一的不一定最相关，Embedding 模型精度有限。

> 🗣️ **大白话：** 初筛出来 20 份简历，再让专家仔细审一遍，按真实相关性重新排名。

### 两种模型对比

| 模型 | 原理 | 速度 | 精度 |
|------|------|------|------|
| **Bi-Encoder（双编码器，即 Embedding 模型）** | 问题和文档分别编码，计算向量相似度 | 快 | 一般 |
| **Cross-Encoder（交叉编码器）** | 问题 + 文档拼在一起输入，直接打分 | 慢 | 高 |

> 🗣️ **大白话：**
> Bi-Encoder：先分别给问题和文档"拍照"，再比较照片像不像
> Cross-Encoder：把问题和文档放一起"面对面比较"，更仔细但更慢

### 常用 Cross-Encoder Reranker 模型

| 模型 | 部署方式 | 特点 |
|------|----------|------|
| **BGE-Reranker-Base** | 本地 | BAAI 出品，中文效果好，轻量，免费 |
| **BGE-Reranker-Large** | 本地 | BGE 大版本，精度更高，显存需求稍大 |
| **BGE-Reranker-V2-M3** | 本地 | 最新版，支持多语言，**推荐首选** |
| **ms-marco-MiniLM** | 本地 | 基于 Microsoft MARCO 数据集，英文效果好，轻量 |
| **ms-marco-electra-base** | 本地 | 基于 ELECTRA 架构，英文精度高 |
| **Jina Reranker V2** | 本地 / API | 支持多语言，开源，社区活跃 |
| **FlashRank** | 本地 | 极致轻量，速度快，适合延迟敏感场景 |
| **Cohere Reranker** | API | 多语言，效果出色，收费，省心之选 |

> 🗣️ **大白话：**
> - 中英文都要强 → 选 **BGE-Reranker-V2-M3**（免费，效果好）
> - 纯英文 + 资源少 → 选 **ms-marco-MiniLM**（更小更快）
> - 不想自己部署 → 直接调 **Cohere API**（效果最好，但收费）

**代码示例：**

```python
# 方式一：本地 BGE-Reranker（推荐）
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-v2-m3", max_length=512)

query = "公司的退款政策是什么？"
candidates = [doc.page_content for doc in retrieved_docs]   # 初步召回的文档

# 批量打分：输入 [(query, doc1), (query, doc2), ...]
pairs = [(query, doc) for doc in candidates]
scores = reranker.predict(pairs)

# 按分数重排，取 Top 3
ranked = sorted(zip(scores, retrieved_docs), key=lambda x: x[0], reverse=True)
top_docs = [doc for _, doc in ranked[:3]]

# 方式二：LangChain + Cohere Reranker（API，更省心）
from langchain_cohere import CohereRerank
from langchain.retrievers.contextual_compression import ContextualCompressionRetriever

reranker = CohereRerank(
    model="rerank-multilingual-v3.0",
    top_n=3,   # 最终保留 Top 3
)

compression_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble_retriever,  # 上一步的混合检索器
)

final_docs = compression_retriever.invoke("公司的退款政策是什么？")
```

## Step 8：组装 Prompt 交给 LLM

**做什么：** 把检索到的文本块 + 用户问题，按模板组成完整 Prompt，交给 LLM 生成答案。

```
【System Prompt（系统提示词）】
你是一个专业的企业知识库助手。
请严格基于以下参考资料回答用户问题。
如果资料中没有相关信息，请明确说"我在知识库中未找到相关信息"，不要编造。

【检索到的参考资料】
资料1：{chunk_1 的内容}
资料2：{chunk_2 的内容}
资料3：{chunk_3 的内容}

【用户问题】
{用户输入的问题}

【要求】
- 回答有依据，可注明来自哪份资料
- 不要超出资料范围发挥
```

**代码示例：**

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_openai import ChatOpenAI

# RAG Prompt 模板
rag_prompt = ChatPromptTemplate.from_messages([
    ("system", """你是一个专业的企业知识库助手。
请严格基于以下参考资料回答用户问题。
如果资料中没有相关信息，请明确说"我在知识库中未找到相关信息"，不要编造。

参考资料：
{context}"""),
    ("human", "{question}"),
])

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def format_docs(docs):
    """将多个文档块拼成字符串，带编号"""
    return "\n\n".join(
        f"[资料{i+1}] {doc.page_content}"
        for i, doc in enumerate(docs)
    )

# 组装完整的 RAG Chain（LCEL 链式语法）
rag_chain = (
    {
        "context": compression_retriever | format_docs,  # 检索 → 格式化
        "question": RunnablePassthrough(),               # 原始问题透传
    }
    | rag_prompt
    | llm
    | StrOutputParser()
)

# 一行调用
answer = rag_chain.invoke("公司的退款政策是什么？")
print(answer)

# 流式输出（适合前端实时显示）
for chunk in rag_chain.stream("公司的退款政策是什么？"):
    print(chunk, end="", flush=True)
```

## Step 9：RAGAS 评估（RAGAS Evaluation）

**RAGAS（Retrieval Augmented Generation Assessment，检索增强生成评估框架）**

> 🗣️ **大白话：** 做完 RAG 系统，怎么知道好不好用？RAGAS 提供 4 个指标，用 LLM 来打分。

---

### 指标一：忠实度（Faithfulness）

**评估：生成的答案有没有胡编乱造？**

> 🗣️ **大白话：** 模型说了 5 句话，有几句是从资料里来的？忠实度越高，"胡说"越少。

**计算方式：**
1. 把答案拆解为若干"原子陈述"（最小的事实单元）
2. 逐条核查每条陈述在检索到的资料里是否有支撑
3. `忠实度 = 有依据的陈述数 / 总陈述数`

**示例：**
> 答案："张三是公司 CEO，负责战略规划，2023 年加入公司"
> → 拆为 3 条陈述，2 条有资料支撑，1 条无依据
> → 忠实度 = 2/3 ≈ **0.67**

---

### 指标二：答案相关性（Answer Relevance）

**评估：答案有没有答非所问？**

> 🗣️ **大白话：** 用答案反推问题，如果推出来的问题和原始问题很像，说明答案切题；反之就是跑题了。

**计算方式：**
1. 把生成的答案交给 LLM，让它根据答案"反推"出 3~5 个可能的问题
2. 计算反推问题与原始问题的向量余弦相似度
3. `答案相关性 = 平均余弦相似度`

---

### 指标三：上下文召回率（Context Recall）

**评估：该找到的资料，有没有都找到？**

> 🗣️ **大白话：** 正确答案需要 5 条线索，你找到了几条？找到 4 条就是 0.8，有漏网的。

**计算方式：**（需要人工标注的标准答案 Ground Truth）
1. 把标准答案拆成若干句子
2. 检查每句话能否在检索到的上下文中找到支撑
3. `上下文召回率 = 有支撑的句子数 / 标准答案总句子数`

**示例：**
> 标准答案有 5 句，检索内容能支撑 4 句
> → 上下文召回率 = 4/5 = **0.80**

---

### 指标四：上下文精确率（Context Precision）

**评估：检索回来的内容，有多少是真正有用的？**

> 🗣️ **大白话：** 你找回来 10 本参考书，其中 7 本根本没用，还干扰了模型判断，精确率就很低。

**计算方式：**
1. 对每个检索到的文本块，判断是否与问题相关
2. `上下文精确率 = 相关块数 / 检索总块数`

**示例：**
> 检索了 5 个块，3 个相关，2 个是噪音
> → 上下文精确率 = 3/5 = **0.60**

---

### 四项指标总览

| 指标 | 评估对象 | 大白话 | 低分原因 |
|------|----------|--------|----------|
| **忠实度** | 生成的答案 | 有没有瞎编 | LLM 幻觉、Prompt 约束不够 |
| **答案相关性** | 生成的答案 | 有没有答非所问 | 检索内容偏题、Prompt 设计差 |
| **上下文召回率** | 检索结果 | 该找的有没有漏 | 分块太小、Embedding 模型弱 |
| **上下文精确率** | 检索结果 | 找回来的有没有废内容 | 召回数量太多、没有 Reranking |

**代码示例：**

## 三、端到端完整流程串联

```python
# ===== 建库阶段 =====
from langchain_community.document_loaders import DirectoryLoader, PyMuPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_chroma import Chroma

# 1. 加载文档
loader = DirectoryLoader("docs/", glob="**/*.pdf", loader_cls=PyMuPDFLoader)
documents = loader.load()

# 2. 切块
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(documents)

# 3. Embedding
embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-large-zh-v1.5")

# 4. 存入向量库
vectorstore = Chroma.from_documents(chunks, embeddings, persist_directory="./db")

# ===== 问答阶段 =====
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever
from sentence_transformers import CrossEncoder
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# 5. 混合检索
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})
bm25_retriever = BM25Retriever.from_documents(chunks, k=10)
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6],
)

# 6. Reranking
reranker_model = CrossEncoder("BAAI/bge-reranker-v2-m3")

def rerank(query, docs, top_n=3):
    pairs = [(query, doc.page_content) for doc in docs]
    scores = reranker_model.predict(pairs)
    ranked = sorted(zip(scores, docs), key=lambda x: x[0], reverse=True)
    return [doc for _, doc in ranked[:top_n]]

# 7. 组装 Prompt + LLM 生成
rag_prompt = ChatPromptTemplate.from_messages([
    ("system", "你是企业知识库助手，请严格基于以下资料回答：\n\n{context}"),
    ("human", "{question}"),
])
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def rag_query(question: str) -> str:
    candidates = ensemble_retriever.invoke(question)
    top_docs = rerank(question, candidates, top_n=3)
    context = "\n\n".join(f"[{i+1}] {d.page_content}" for i, d in enumerate(top_docs))
    chain = rag_prompt | llm | StrOutputParser()
    return chain.invoke({"context": context, "question": question})

print(rag_query("公司的退款政策是什么？"))
```

## 四、进阶方向

| 方向 | 说明 |
|------|------|
| **GraphRAG** | 把文档构建成知识图谱，支持多跳推理 |
| **Self-RAG** | 模型自己判断是否需要检索，动态决策 |
| **Agentic RAG** | 用 Agent 编排多步检索，处理复杂问题 |
| **Long-Context RAG** | 利用超长上下文模型（如 Gemini 1M token）减少分块 |
| **RAG + 微调** | 用 RAG 数据微调小模型，降低推理成本 |
