# RAG 知识体系

## 一、什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）**

### 为什么需要 RAG？

LLM 有两个天生缺陷：
- **训练数据有截止日期**：不知道最新发生的事情
- **无法接触私有数据**：公司内部文档没法用来训练模型

RAG 的解法：把私有文档存进向量数据库，用户提问时先检索相关文本块，再把问题 + 文本块一起交给 LLM 生成答案。

---

## 二、RAG 全流程

整体分两个阶段：**建库** + **问答**

```
【建库阶段】 文档加载 → 文档切块 → 向量化 → 存入向量数据库

【问答阶段】 查询优化 → 混合检索 → Reranking重排 → 组装Prompt → LLM生成 → RAGAS评估
```

---

## Step 1：文档加载（Document Loading）

**做什么：** 把各种格式的文件，统一转换成纯文本。

支持格式：PDF、Word、Excel、CSV 等

**工具选择：**

| 方式 | 工具 |
|------|------|
| 框架组件 | LangChain / LlamaIndex 的 Document Loaders |
| 开源库 | PyMuPDF（PDF）、python-docx（Word）、pandas（CSV） |
| 高阶处理 | OCR + LLM + Prompt，处理扫描件、复杂表格 |

---

## Step 2：文档切块（Chunking）

**做什么：** 把长文本切成小块，因为 LLM 上下文窗口有限，不能把整本书全塞进去。

> 🗣️ **大白话：** 块太小 → 没上下文，答不好；块太大 → 噪音多，检索不准还浪费 Token。

### 四种切块策略

| 策略 | 说明 |
|------|------|
| **固定长度切块** | 按固定 Token 数/字符数切分，简单高效，比如每块 500 字 |
| **语义切块** | 按段落、章节、标题等语义边界切分，保持信息完整 |
| **父子切块（Parent-Child Chunking）** | 小块（256字）用于检索，命中后返回对应大块（1024字）给 LLM |
| **滑动窗口切块** | 固定长度基础上加重叠部分（如 100 字），避免切断关键信息，提升召回 |

---

## Step 3：向量化（Embedding）

**做什么：** 把文本块转成向量（一串数字），让计算机能做语义检索。

> 🗣️ **大白话：** 把"苹果"和"水果"都变成空间中的点，语义越接近的词，点的距离越近。

### 常用 Embedding 模型

| 模型 | 部署方式 | 特点 |
|------|----------|------|
| **BGE**（BAAI General Embedding，北京智源通用向量模型） | 本地 | 中文效果最好，免费，首选 |
| **text2vec** | 本地 | 轻量，适合资源有限的环境 |
| **OpenAI text-embedding-3** | API | 效果好，收费 |
| **Cohere Embed** | API | 多语言强，收费 |

> BGE 由北京人工智能研究院（BAAI，Beijing Academy of Artificial Intelligence）开源。

---

## Step 4：向量数据库（Vector Database）

**做什么：** 存储所有向量，并根据相似度快速检索。

### 相似度计算方式

| 方式 | 原理 | 越相似则 |
|------|------|----------|
| **余弦相似度（Cosine Similarity）** | 衡量两向量夹角 | 值越接近 1 |
| **欧氏距离（Euclidean Distance）** | 衡量两向量之间的直线距离 | 值越小 |
| **内积（Dot Product）** | 两向量的点积 | 值越大 |

### 常用向量数据库

| 数据库 | 适用场景 |
|--------|----------|
| **ChromaDB** | 学习 / 原型开发，易上手 |
| **PGVector** | 已有 PostgreSQL，直接扩展，减少系统复杂度 |
| **Milvus** | 大规模数据，支持分布式，性能强但配置复杂 |
| **Qdrant** | 生产环境，高维向量，性能优越 |

---

## Step 5：查询优化（Query Optimization）

**为什么要优化：** 用户提问往往口语化、模糊，直接检索效果差。

> 🗣️ **大白话：** 用户说"RAG效果不好咋整"，直接搜这句话检索不到什么，但改成"RAG检索准确率的影响因素及优化方法"就能搜到很多有用内容。

### 三种优化技巧

**① Query Rewriting（查询重写）**
用规则或模型将用户问题改写成更适合检索的表达。
> 示例："我想看公司财报" → "公司 2023 年财报摘要"

**② HyDE（Hypothetical Document Embedding，假设文档向量化）**
让 LLM 先根据问题生成一段"假设的答案文档"，再对这段假设文档做 Embedding 检索，最终用检索到的真实文档 + 原始问题回答。
> 🗣️ **大白话：** 先"脑补"一个答案，用这个答案去找资料，比直接用问题找更准。

**③ 多查询并行（Multi-Query）**
针对同一问题从多个角度改写，并行检索，合并结果，提升召回率。
> 示例："RAG 和微调怎么选？"拆为：
> - 角度1：RAG 的局限性有哪些？
> - 角度2：微调的使用场景是什么？
> - 角度3：微调 vs RAG 对比分析

---

## Step 6：混合检索（Hybrid Retrieval）

**做什么：** 结合向量检索（语义）和关键词检索（精确），取两者之长。

| 检索方式 | 优点 | 缺点 |
|----------|------|------|
| **向量检索**（语义检索） | 理解语义，能处理模糊问题 | 精确词匹配弱 |
| **关键词检索**（如 BM25） | 精确匹配关键词 | 不理解语义 |
| **混合检索** | 结合两者，召回更全 | 需要合并排序 |

> 🗣️ **大白话：** 一个靠"意思"找，一个靠"关键词"找，两种一起用，才能又准又全。

---

## Step 7：重排序（Reranking）

**为什么要重排：** 混合检索召回了 Top 20，但排第一的不一定最相关，Embedding 模型精度有限。

> 🗣️ **大白话：** 初筛出来 20 份简历，再让专家仔细审一遍，按真实相关性重新排名。

### 两种模型对比

| 模型 | 原理 | 速度 | 精度 |
|------|------|------|------|
| **Bi-Encoder（双编码器，即 Embedding 模型）** | 问题和文档分别编码，计算向量相似度 | 快 | 一般 |
| **Cross-Encoder（交叉编码器）** | 问题 + 文档拼在一起输入，直接打分 | 慢 | 高 |

> 🗣️ **大白话：**
> Bi-Encoder：先分别给问题和文档"拍照"，再比较照片像不像
> Cross-Encoder：把问题和文档放一起"面对面比较"，更仔细但更慢

### 常用 Cross-Encoder Reranker 模型

| 模型 | 部署方式 | 特点 |
|------|----------|------|
| **BGE-Reranker-Base** | 本地 | BAAI 出品，中文效果好，轻量，免费 |
| **BGE-Reranker-Large** | 本地 | BGE 大版本，精度更高，显存需求稍大 |
| **BGE-Reranker-V2-M3** | 本地 | 最新版，支持多语言，**推荐首选** |
| **ms-marco-MiniLM** | 本地 | 基于 Microsoft MARCO 数据集，英文效果好，轻量 |
| **ms-marco-electra-base** | 本地 | 基于 ELECTRA 架构，英文精度高 |
| **Jina Reranker V2** | 本地 / API | 支持多语言，开源，社区活跃 |
| **FlashRank** | 本地 | 极致轻量，速度快，适合延迟敏感场景 |
| **Cohere Reranker** | API | 多语言，效果出色，收费，省心之选 |

> 🗣️ **大白话：**
> - 中英文都要强 → 选 **BGE-Reranker-V2-M3**（免费，效果好）
> - 纯英文 + 资源少 → 选 **ms-marco-MiniLM**（更小更快）
> - 不想自己部署 → 直接调 **Cohere API**（效果最好，但收费）

---

## Step 8：组装 Prompt 交给 LLM

**做什么：** 把检索到的文本块 + 用户问题，按模板组成完整 Prompt，交给 LLM 生成答案。

```
【System Prompt（系统提示词）】
你是一个专业的企业知识库助手。
请严格基于以下参考资料回答用户问题。
如果资料中没有相关信息，请明确说"我在知识库中未找到相关信息"，不要编造。

【检索到的参考资料】
资料1：{chunk_1 的内容}
资料2：{chunk_2 的内容}
资料3：{chunk_3 的内容}

【用户问题】
{用户输入的问题}

【要求】
- 回答有依据，可注明来自哪份资料
- 不要超出资料范围发挥
```

---

## Step 9：RAGAS 评估（RAGAS Evaluation）

**RAGAS（Retrieval Augmented Generation Assessment，检索增强生成评估框架）**

> 🗣️ **大白话：** 做完 RAG 系统，怎么知道好不好用？RAGAS 提供 4 个指标，用 LLM 来打分。

---

### 指标一：忠实度（Faithfulness）

**评估：生成的答案有没有胡编乱造？**

> 🗣️ **大白话：** 模型说了 5 句话，有几句是从资料里来的？忠实度越高，"胡说"越少。

**计算方式：**
1. 把答案拆解为若干"原子陈述"（最小的事实单元）
2. 逐条核查每条陈述在检索到的资料里是否有支撑
3. `忠实度 = 有依据的陈述数 / 总陈述数`

**示例：**
> 答案："张三是公司 CEO，负责战略规划，2023 年加入公司"
> → 拆为 3 条陈述，2 条有资料支撑，1 条无依据
> → 忠实度 = 2/3 ≈ **0.67**

---

### 指标二：答案相关性（Answer Relevance）

**评估：答案有没有答非所问？**

> 🗣️ **大白话：** 用答案反推问题，如果推出来的问题和原始问题很像，说明答案切题；反之就是跑题了。

**计算方式：**
1. 把生成的答案交给 LLM，让它根据答案"反推"出 3~5 个可能的问题
2. 计算反推问题与原始问题的向量余弦相似度
3. `答案相关性 = 平均余弦相似度`

---

### 指标三：上下文召回率（Context Recall）

**评估：该找到的资料，有没有都找到？**

> 🗣️ **大白话：** 正确答案需要 5 条线索，你找到了几条？找到 4 条就是 0.8，有漏网的。

**计算方式：**（需要人工标注的标准答案 Ground Truth）
1. 把标准答案拆成若干句子
2. 检查每句话能否在检索到的上下文中找到支撑
3. `上下文召回率 = 有支撑的句子数 / 标准答案总句子数`

**示例：**
> 标准答案有 5 句，检索内容能支撑 4 句
> → 上下文召回率 = 4/5 = **0.80**

---

### 指标四：上下文精确率（Context Precision）

**评估：检索回来的内容，有多少是真正有用的？**

> 🗣️ **大白话：** 你找回来 10 本参考书，其中 7 本根本没用，还干扰了模型判断，精确率就很低。

**计算方式：**
1. 对每个检索到的文本块，判断是否与问题相关
2. `上下文精确率 = 相关块数 / 检索总块数`

**示例：**
> 检索了 5 个块，3 个相关，2 个是噪音
> → 上下文精确率 = 3/5 = **0.60**

---

### 四项指标总览

| 指标 | 评估对象 | 大白话 | 低分原因 |
|------|----------|--------|----------|
| **忠实度** | 生成的答案 | 有没有瞎编 | LLM 幻觉、Prompt 约束不够 |
| **答案相关性** | 生成的答案 | 有没有答非所问 | 检索内容偏题、Prompt 设计差 |
| **上下文召回率** | 检索结果 | 该找的有没有漏 | 分块太小、Embedding 模型弱 |
| **上下文精确率** | 检索结果 | 找回来的有没有废内容 | 召回数量太多、没有 Reranking |
